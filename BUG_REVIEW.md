# דוח בדיקת קוד – Gym Tracker

תאריך בדיקה: 2026-03-22

## קריטי

### 1) מחיקות גלובליות ב-Supabase ללא סקופ משתמש (סיכון לאובדן מידע לכלל המשתמשים)
**איפה:**
- `saveSettingsToSupabase()` מוחק את כל `user_settings` ואז מכניס רשומה חדשה.
- `saveWorkoutsToSupabase()` מוחק את כל `workout_plans` ואז מכניס מחדש.
- `saveWeightGoal()` מוחק את כל `weight_goal` ואז מכניס מחדש.

**השפעה:** משתמש אחד עלול למחוק נתונים של אחרים (או של עצמו במקביל מסשן אחר), במיוחד אם אין RLS קשיח.

**טיפול מוצע:**
1. להוסיף `user_id` לכל טבלה רלוונטית ולסנן כל קריאה לפי `eq.<user_id>`.
2. להחליף דפוס `DELETE all + INSERT` ב-`upsert` (או `PATCH` ממוקד לפי מזהה רשומה/משתמש).
3. להגדיר RLS ב-Supabase כך שמשתמש יכול לגשת רק לשורות שלו.

## גבוה

### 2) XSS מבוסס נתוני משתמש דרך `innerHTML`/תבניות מחרוזת
**איפה:**
- שמות תרגילים מוזרקים ל-HTML ול-`onclick` ללא escaping.
- הערת אימון (`session.note`) מוצגת ישירות ב-HTML.
- `workoutNote` מוזרק לתוך `<textarea>...</textarea>` בתוך תבנית HTML.

**השפעה:** תוקף יכול להזריק JS זדוני באמצעות שם תרגיל/הערה, ולהריץ קוד בדפדפן (גניבת נתונים, שיבוש UI, פעולות לא מורשות).

**טיפול מוצע:**
1. לא להשתמש ב-`innerHTML` לערכים שמגיעים ממשתמש; להשתמש ב-`textContent`/`setAttribute`.
2. להסיר inline handlers (`onclick="..."`) ולעבור ל-`addEventListener`.
3. כשאין ברירה, להשתמש בפונקציית escape ל-HTML + escape ל-context של JS-string.

### 3) חשיפת מפתח Supabase בקוד לקוח
**איפה:** `config.js` כולל `SUPABASE_KEY` גלוי.

**השפעה:** אם ההרשאות בבסיס הנתונים לא קשיחות מספיק, כל אחד יכול לבצע קריאות API ישירות.

**טיפול מוצע:**
1. לוודא שזהו publishable/anon key בלבד (לא service role).
2. להקשיח RLS ברמת טבלה ו-policy.
3. פעולות רגישות לבצע דרך backend/edge function עם אימות משתמש.

## בינוני

### 4) אובדן שדה `note` בסנכרון סשנים
**איפה:**
- בעת שמירה ל-Supabase לא נשלח `note`.
- בעת טעינה מ-Supabase לא ממופים `note` חזרה ל-state.

**השפעה:** הערות אימון קיימות לוקאלית אבל הולכות לאיבוד אחרי רענון/טעינה מהשרת.

**טיפול מוצע:**
1. להוסיף עמודת `note` ב-`workout_sessions`.
2. לשלוח `note` ב-`saveSessionToSupabase` ולמפות ב-`loadSessionsFromSupabase`.
3. להוסיף migration/ברירת מחדל ריקה לרשומות ישנות.

### 5) שינוי שם תרגיל לא מעדכן תרגילים בפורמט legacy (string)
**איפה:** `updateLibraryEntry()` מעדכן רק אובייקטי exercise; אם exercise הוא string הוא לא מתעדכן.

**השפעה:** נוצר חוסר עקביות: הספרייה מציגה שם חדש, אך אימונים ישנים ממשיכים להצביע על שם ישן.

**טיפול מוצע:**
1. בזמן עדכון שם, לטפל גם במקרה `typeof ex === "string"` ולהחליף את הערך.
2. להריץ migration חד-פעמי שממיר תרגילים לייצוג אחיד כאובייקט `{name, rest, category}`.

## המלצות רוחביות
- להוסיף בדיקות אינטגרציה קצרות לשמירה/טעינה/שינוי שם תרגיל.
- להוסיף שכבת `sanitize` מרכזית לכל טקסט שמגיע ממשתמש לפני render.
- להפעיל lint rule שאוסר inline event handlers ו-`innerHTML` לא מבוקר.
